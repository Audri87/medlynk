# SD-004 — Clinical Platform Context Map

**Status**: Release v1.0

---

## Purpose

Define the strategic relationships between the certified Bounded Contexts of the Clinical Platform and describe how information flows across context boundaries.

The Context Map documents collaboration, dependencies and integration patterns.

It does not redefine business ownership.

---

## 1. Certified Bounded Contexts

The Clinical Platform contains two Core Bounded Contexts.

| Bounded Context | Mission |
|---|---|
| Clinical Work | Coordinate clinical work and preserve Clinical Responsibility throughout the Clinical Activity lifecycle. |
| Clinical Knowledge | Preserve validated clinical knowledge generated by clinical work. |

These ownership boundaries are defined in SD-003.

---

## 2. Context Relationships

### Clinical Work → Clinical Knowledge

**Relationship Pattern**

Customer / Supplier

**Supplier**

Clinical Work owns the production of validated Clinical Contributions.

**Customer**

Clinical Knowledge consumes validated Clinical Contributions and transforms them into persistent clinical knowledge.

---

## 3. Integration Model

Communication between Bounded Contexts shall occur through Domain Events.

No Aggregate or Domain Object shall cross a Bounded Context boundary.

| Source | Event | Target |
|---|---|---|
| Clinical Work | ClinicalContributionValidated | Clinical Knowledge |

---

## 4. Published Events

Clinical Knowledge publishes events consumed by architectural components.

| Published Event | Consumers |
|---|---|
| ClinicalContributionPublished | Workspace |
| ClinicalContributionPublished | Search / Read Models |
| ClinicalContributionPublished | Business Widgets |

These consumers are architectural components and not Bounded Contexts.

### Read Model Flow

`ClinicalContributionPublished` is the entry point for all read-side projections within the Clinical Platform.

- Clinical Knowledge publishes `ClinicalContributionPublished` each time a Clinical Contribution is made available.
- Read Models subscribe to this event and build optimized projections — Search index, Care Record timeline, Patient overview.
- Workspace consumes Read Models exclusively.
- Workspace shall never query Clinical Knowledge directly.
- No read path shall bypass the Read Model layer to access a Core Bounded Context.

---

## 5. Architectural Components

The following components collaborate with the Core Bounded Contexts but do not own business models.

| Component | Role |
|---|---|
| Clinical Workspace | UI Composition Layer |
| Search / Discovery | Read Model |
| Business Widgets | Presentation Components |
| External Clinical Exchange | Anti-Corruption Layer |
| Decision Support | Application Service |
| Governance | Cross-cutting Concern |

---

## 6. Dependency Rules

The following dependency rules are mandatory.

**Rule 1**

Clinical Knowledge shall never modify Clinical Work.

**Rule 2**

Workspace shall never contain business logic.

**Rule 3**

Business Widgets shall never own business rules.

**Rule 4**

Search shall never become the source of truth.

**Rule 5**

External systems shall communicate through the Anti-Corruption Layer.

**Rule 6**

Clinical Work shall never read directly from Clinical Knowledge.

Whenever historical clinical information is required, Clinical Work shall access dedicated Read Models or Application Services rather than querying the Clinical Knowledge Domain Model.

## 8. Strategic Consequences

The Clinical Platform is intentionally centred around two Core Bounded Contexts.

Supporting architectural components consume information without becoming business owners.

This preserves a single source of truth for each business concept.

Clinical Continuity emerges from the collaboration between the Core Bounded Contexts rather than from any individual component.
